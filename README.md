# Система бронирования ресторана (учебный проект)

Мини-приложение для управления бронированиями столиков в ресторане.
Построено на **PostgreSQL** + **Tkinter** без ORM — все запросы пишутся напрямую через собственный драйвер.

## Возможности

- **Пользователи** — создание, поиск, редактирование, мягкое удаление (активация/деактивация). Роли: клиент, администратор.
- **Столы** — управление номером, вместимостью, расположением (зал / терраса / VIP / бар), доступностью.
- **Бронирования** — двухшаговый процесс: создание со статусом «Ожидает» → проверка доступности и подтверждение. Фильтрация по статусу и дате. Детали с JOIN-запросом.
- **Проверка доступности** — отдельная вкладка: показывает, свободен ли стол на выбранное время, и выводит список существующих бронирований.
- **Контроль мест** — при превышении числа гостей над вместимостью стола выводится предупреждение, а в примечания бронирования добавляется автоматическая пометка.

## Требования

- Python 3.10+
- PostgreSQL 13+
- Зависимости: `psycopg2-binary`, `python-dotenv` (tkinter входит в стандартную библиотеку)

## Установка и запуск

### 1. Клонировать репозиторий

```bash
git clone <url-репозитория>
cd <папка-проекта>
```

### 2. Создать виртуальное окружение и установить зависимости

```bash
python -m venv venv

# Windows
venv\Scripts\activate

# Linux / macOS
source venv/bin/activate

pip install -r requirements.txt
```

### 3. Настроить подключение к БД

Скопировать `.env.example` в `.env` и заполнить параметры:

```bash
cp .env.example .env
```

```env
DB_NAME=restaurant
DB_USER=postgres
DB_PASSWORD=your_password
DB_HOST=localhost
DB_PORT=5432
```

### 4. Запустить приложение

```bash
python app.py
```

Таблицы в БД создаются автоматически при первом запуске.

## Архитектура

### Взаимодействие слоёв

```
┌─────────────────────────────────┐
│          app.py (GUI)           │   Tkinter, 4 вкладки
├─────────────────────────────────┤
│        backend.py (сервис)      │   Бизнес-логика, валидация
├──────────┬──────────┬───────────┤
│  user.py │ tables.py│ booking.py│   Модели (статические методы)
├──────────┴──────────┴───────────┤
│    postgres_driver.py (драйвер) │   PostgreSQL CRUD
└─────────────────────────────────┘
```

- **GUI** (`app.py`) — запускает операции БД в отдельных потоках, чтобы не блокировать интерфейс.
- **Сервисный слой** (`backend.py`) — каждая функция открывает и закрывает собственное соединение с БД.
- **Модели** (`models/`) — статические методы, принимают экземпляр `PostgresDriver`. Содержат SQL и десериализацию.
- **Драйвер** (`postgres_driver.py`) — обёртка над `psycopg2` с контекстным менеджером, CRUD-методами и поддержкой транзакций.

## Вкладки GUI

### Пользователи

Создание, поиск по ID и email, редактирование, удаление, активация/деактивация. Выбор строки в таблице заполняет форму.

### Столы

Создание, поиск по ID и номеру, редактирование, удаление, управление доступностью. Расположение: зал, терраса, VIP, бар.

### Бронирования

| Кнопка | Действие |
|---|---|
| **Создать** | Создаёт бронирование со статусом «Ожидает» |
| **Найти по ID** | Заполняет форму данными бронирования |
| **💾 Сохранить** | Сохраняет изменения существующего бронирования |
| **Удалить** | Удаляет бронирование с подтверждением |
| **Подробности (JOIN)** | Показывает полную информацию: пользователь + стол |
| **✓ Проверить и подтвердить** | Проверяет пересечение времени и подтверждает, либо отказывает |
| **✗ Отменить** | Статус → «Отменено» |
| **✔ Завершить** | Статус → «Завершено» |

Пользователь и стол выбираются из выпадающих списков. Доступна фильтрация по статусу и дате.

## Двухшаговый процесс бронирования

1. **Создание** — бронирование сохраняется со статусом `pending` без проверки пересечений.
2. **Проверка и подтверждение** — кнопка «✓ Проверить и подтвердить» проверяет, не занято ли время. Если свободно — статус меняется на `confirmed`. Если есть пересечение — статус остаётся `pending`, и пользователь может изменить данные.

## Контроль местности

При создании или обновлении бронирования система проверяет: если число гостей превышает вместимость выбранного стола:

1. Выводится предупреждение `⚠ Несоответствие мест`.
2. В примечания автоматически добавляется пометка: `[Мест: X, гостей: Y]`.

Бронирование при этом не блокируется — решение остаётся за оператором.
